1. Resumen Ejecutivo

1.1 Alcance del informe

Este documento describe el estado técnico completo del Sistema de Créditos IPSFA tras la serie de mejoras implementadas desde mayo hasta julio de 2026 (versiones v6.0 a v6.7). Cubre arquitectura, base de datos, API REST, seguridad, frontend, refactorizaciones y rediseños de interfaz.

El sistema gestiona créditos de equipos para personal militar a través de tres tiendas (Caracas, Maracay, Maracaibo), con roles de administrador y operador, autenticación JWT, conciliación bancaria, generación de reportes y seguimiento de actividades.

2.3 Flujo de autenticación

  1. Usuario ingresa credenciales en login.html
  2. POST /api/auth/login → verifica bcrypt + IP real del socket
  3. Backend genera JWT con {id, usuario, rol, tienda, token_version}
  4. Token se almacena en localStorage + variable global window.token
  5. Panel.js monta interceptor: todas las llamadas fetch envían Authorization: Bearer <token>
  6. Middleware verificarToken valida firma + busca token_version en BD (revocación)
  7. Logout: POST /api/auth/logout → incrementa token_version en BD → token muere en todas las pestañas
Nota de seguridad: La revocación funciona porque el token JWT incluye token_version. Al hacer logout, el backend incrementa este valor en la tabla usuarios. Cualquier token con la versión antigua es rechazado, incluso en pestañas abiertas o tokens robados.

6.2 JWT + revocación

Header:  { alg: "HS256", typ: "JWT" }
Payload: { 
  id: 1, 
  usuario: "admin", 
  rol: "administrador", 
  tienda: "caracas",
  token_version: 3, 
  iat: 1750000000, 
  exp: 1750032800  // 8 horas
}
  • Firma: HS256 con JWT_SECRET de 64+ caracteres aleatorios (rotado en v6.1)
  • Expiración: 8 horas (configurable en .env)
  • Revocación: Al logout, UPDATE usuarios SET token_version = token_version + 1 WHERE id = ?. Cualquier token con versión anterior es rechazado.
  • Rate limit: 5 intentos de login cada 15 minutos por IP

8.2 Patrón genérico

La clave de la refactorización fue reemplazar 3 archivos SPA casi idénticos (solo cambiaban sufijos de IDs y endpoints) por una clase TiendaApp que recibe su configuración al instanciarse:

// Antes: tienda-caracas-spa.js, tienda-maracay-spa.js, tienda-maracaibo-spa.js
// (~1.500 líneas cada uno, 61 funciones duplicadas)

// Después: tienda-spa.js (2.876 líneas)

  2. HTTPS: Implementar con certificado (Let's Encrypt para dominio público, o autofirmado para red local)

  4. Backup: Tarea programada diaria de pg_dump con retención de 30 días

  6. Monitorización: PM2 para reinicio automático si el proceso cae
